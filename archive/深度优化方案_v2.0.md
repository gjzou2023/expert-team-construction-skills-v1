# 全域专家团构建skills v1.4.0 深度优化方案

> 基于系统思维·结构化思维·后退式策略三维分析 | 目标版本: v2.0.0

---

## 一、系统思维分析：将专家团构建视为完整系统

### 1.1 系统边界与子系统识别

当前 v1.4.0 的 5 层 39-Skill 架构可分解为 **4 个核心子系统** + **1 个横贯子系统**：

| 子系统 | 包含组件 | 核心功能 | 当前状态评估 |
|--------|---------|---------|------------|
| **需求解析子系统** | S1需求深潜 + core-mental-model-engine + core-domain-classifier + core-complexity-channel-selector | 理解用户真实需求并确定领域与通道 | 🟡 通道判定与领域分类耦合度偏高，导致早期锁定风险 |
| **架构设计子系统** | S3链路拆解 + S4交付物锚定 + S5架构设计 + core-deliverable-backward-engine | 从需求推导出可执行架构 | 🟡 S3/S4 逻辑重叠，快速通道跳过逻辑分散 |
| **交付物化子系统** | S6工具链匹配 + S7专家包生成 + S8平台执行 + 全部 platform-*-adapter | 将架构转化为可导入平台的产品 | 🟢 核心链路完整，但平台适配层差异化不足 |
| **保障子系统** | 全部 protocol-* + constraint-* + core-state-management-engine + core-symbol-system | 全程质量/合规/安全/反馈保障 | 🟡 协议间调用关系复杂，存在冗余检查点 |
| **横贯约束子系统** | L4 全局约束层 (5个 constraint-*) | 全程横贯规则引擎 | 🟡 20+11=31 条规则缺少优先级冲突消解机制 |

### 1.2 关键反馈回路识别

#### 正反馈回路（增强型）

```
用户确认 → 领域确信度↑ → 通道升级(快速→标准→严格) → 信息采集更完整 → 匹配精度↑ → 用户满意度↑
```

**问题**：当前 v1.4.0 中，这个正反馈回路是隐式的，没有显式的"确信度累计"机制。S1 的置信度评估结果没有跨阶段传递，S2 消歧后的 confidence 也没有影响后续 S3-S5 的行为。

#### 负反馈回路（纠偏型）

```
质量门控不通过 → 回退到前序阶段 → 修正产出 → 重新验证
```

**问题**：当前回退逻辑（protocol-confirmation-node + core-state-management-engine）仅支持单阶段回退，不支持跨阶段回退。例如 S7 质量门控发现 S4 交付物锚定有误时，没有路径直接回退到 S4。

#### 延迟反馈回路（迭代型）

```
S8上线后 → 用户使用 → 反馈数据 → v2.0迭代 → 重新触发S1
```

**问题**：当前版本迭代触发信号仅有4种文字描述（"做不了X"/"想要新能力"/"我想升级"/核心指标低于阈值），没有量化指标体系和自动触发机制。

### 1.3 系统瓶颈定位

| 瓶颈 | 位置 | 影响 | 严重度 |
|------|------|------|--------|
| **B1: 需求-领域强耦合** | S1→S2 | 早期通道判定过度依赖领域分类结果，一旦领域分类偏差，整条链路偏移 | 🔴 P0 |
| **B2: S3/S4 逻辑重叠** | S3+S4 | 链路拆解与交付物锚定有约40%重复推理，快速通道需跳过S3但S4又依赖S3输出 | 🟡 P1 |
| **B3: 协议调用爆炸** | S7汇聚点 | S7同时调用全部L2协议（QG+CE+DS+EH+FL+HA+KP+AT+SQG+CN=10个），延迟和Token消耗巨大 | 🔴 P0 |
| **B4: 状态传递断层** | 跨阶段 | 除 core-state-management-engine 的快照机制外，没有显式的"上下文背包"传递关键决策理由 | 🟡 P1 |
| **B5: 平台适配浅层化** | L3层 | 9个适配器中仅 WorkBuddy 有详细执行逻辑，其余8个仅生成配置文件模板 | 🟡 P1 |
| **B6: 约束冲突无仲裁** | L4层 | 31条规则（20强制+11灵活）之间可能存在冲突，缺少仲裁机制 | 🟡 P1 |

### 1.4 系统优化杠杆点

基于 Donella Meadows 的杠杆点框架，按影响力从低到高排列：

| 杠杆点 | 类型 | 具体改进 | 预期效果 |
|--------|------|---------|---------|
| **LP1: 参数调整** | 低杠杆 | 调整S1问题数量、MECE检查阈值 | 局部优化 |
| **LP2: 缓冲区大小** | 中杠杆 | 在S2-S3间增加"确认缓冲区"，允许用户补充信息 | 减少回退 |
| **LP3: 反馈回路强化** | 中杠杆 | 增加确信度累计机制、跨阶段回退路径 | 提高纠偏效率 |
| **LP4: 信息流重构** | 高杠杆 | 引入"上下文背包"贯穿S1-S8 | 消除状态断层 |
| **LP5: 规则演进** | 高杠杆 | 约束冲突仲裁机制 + 规则优先级矩阵 | 消除规则矛盾 |
| **LP6: 系统目标重定义** | 最高杠杆 | 从"生成专家包"升级为"持续匹配用户需求与专家能力" | 支持迭代进化 |

## 二、结构化思维分析：能力层级指标体系

### 2.1 四维量化指标体系

将专家团构建能力拆解为4个可量化的层级指标：

#### 维度1: 需求理解精度 (Requirement Understanding Precision, RUP)

| 指标 | 当前基线 | v2.0目标 | 测量方法 |
|------|---------|---------|---------|
| 首轮需求捕获率 | ~60% (S1的9个问题无法覆盖所有场景) | ≥80% | 用户回答"不适用"的问题占比 |
| 歧义消解完成率 | ~70% (四维消解模板存在，但触发条件不够精确) | ≥90% | S2输出中 disambiguation_log 条目与四维度匹配率 |
| 需求变更吸收率 | ~40% (S7后的变更需重新走全流程) | ≥70% | 变更请求到完成修正的平均阶段回退数 |
| 上下文传递完整率 | ~50% (跨阶段仅传递结构化字段，不传递决策理由) | ≥85% | 下游阶段能复述上游关键决策理由的比例 |

#### 维度2: 专家覆盖广度 (Expert Coverage Breadth, ECB)

| 指标 | 当前基线 | v2.0目标 | 测量方法 |
|------|---------|---------|---------|
| 领域类型覆盖率 | 6类(A-F) | 6类+混合组合+时序演化 | S2输出中 domain_profile 的灵活性 |
| 行业知识骨架数 | 8个(已扩展) | 8+行业子领域图谱(可动态扩展) | core-domain-classifier 可识别的行业子领域数 |
| 平台适配覆盖率 | 9平台(但8个仅模板) | 9平台均含完整执行逻辑 | 每个platform-adapter的SKILL.md行数 |
| 角色模板库规模 | 无(每次从零生成) | ≥30个预置角色模板(按领域+平台) | S5可引用的预置角色模板数 |

#### 维度3: 匹配深度 (Matching Depth, MD)

| 指标 | 当前基线 | v2.0目标 | 测量方法 |
|------|---------|---------|---------|
| 角色-交付物映射精度 | MECE三问(定性) | MECE三问+覆盖率量化指标(定量) | S5输出的MECE量化得分 |
| 工具-能力匹配精度 | 4级优先级+5维对比 | 4级优先级+7维对比(新增"合规适配度"和"学习曲线") | S6输出的工具推荐精准度 |
| SOP编排复杂度适配 | 3通道(fast/standard/strict) | 3通道+2模式(首次构建/迭代优化) | S5-SOP可处理的分支场景数 |
| 反馈回路适配深度 | 按领域激活1-3个 | 按领域×通道×平台矩阵激活 | 每个组合的反馈回路配置完整度 |

#### 维度4: 场景适配度 (Scenario Adaptation, SA)

| 指标 | 当前基线 | v2.0目标 | 测量方法 |
|------|---------|---------|---------|
| 首次构建成功率 | 未知(无数据) | ≥85% | S8 delivery_status="ready_for_use"的比例 |
| 迭代优化触达率 | 4种文字触发信号 | 4种信号+量化指标+自动触发 | 自动化迭代触发的场景覆盖率 |
| 紧急终止可用率 | 快速模式最小输出填充 | 快速模式+合规红线保留+质量标注 | 紧急终止输出的可用交付物比例 |
| 跨平台迁移成功率 | 无(无迁移指南) | ≥70% | 从平台A迁移到平台B的配置转换成功率 |

### 2.2 输入→输出的完整逻辑链路

```
输入(用户原始需求)
  ↓ [RUP 需求理解精度]
S1 需求深潜 → 需求画像(core_need + keywords + constraints + channel_hint)
  ↓ [RUP 歧义消解]
S2 领域消歧 → 领域画像(domain_profile + compliance_activation_map + feedback_loops)
  ↓ [ECB 专家覆盖广度]
S3 链路拆解 → 链路图谱(chain_nodes + breakpoints + dependencies)
  ↓ [MD 匹配深度]
S4 交付物锚定 → 交付物清单(deliverables + cross_channel_sharing + platform)
  ↓ [MD 角色-交付物映射]
S5 架构设计 → 角色架构(roles + sop + feedback_design + compliance_design)
  ↓ [MD 工具-能力匹配]
S6 工具链匹配 → 工具矩阵(tool_matrix + degradation_plan + security_assessment)
  ↓ [SA 场景适配度]
S7 专家包生成 → 完整专家包(plugin.json + agents + settings + references)
  ↓ [SA 平台适配]
S8 平台执行 → 交付确认(execution_result + delivery_confirmation + rollback_snapshot)
  ↓
输出(可导入平台的专家团)
```

**关键断裂点**：S2→S3 之间缺少"架构可行性预检"；S5→S6 之间缺少"工具生态探测"；S7→S8 之间缺少"平台兼容性预校验"。

## 三、后退式策略分析：从目标倒推缺陷

### 3.1 最终目标定义

**最终目标**：用户需求与专家能力的精准匹配，以可量化指标衡量——

> 当用户表达一个需求时，系统能在最小交互成本下，生成一个在该用户的目标平台上可直接运行、持续进化、合规安全的专家团。

### 3.2 倒推路径与缺陷发现

#### 第一步倒推：从"可直接运行"倒推

**前提条件**：S8 的 execution_result 必须包含 "ready_for_use" 状态。

**当前缺陷**：

1. **D1: 平台执行验证不可自动化** — S8 的5项检查（功能完整性/合规性/数据安全/角色协作/降级方案）在当前框架下是伪代码描述，无法在真实平台环境中自动执行。WorkBuddy 的 `workbuddy_create_agent` 函数是假设的，没有真实 API 对接。

2. **D2: 8个平台适配器仅输出配置模板** — 除 WorkBuddy 外，Codex/Hermes/飞书/n8n/ComfyUI/Coze/Dify 适配器只有框架性描述，缺少平台特有的字段映射、格式转换和导入步骤。

3. **D3: 回滚机制不可操作** — rollback_snapshot 仅有 snapshot_id 和 rollback_to 描述，没有实际的版本管理机制（如 git commit hash、文件哈希校验）。

**改进优先级**：P0（阻塞性缺陷，专家包无法真正"运行"）

#### 第二步倒推：从"持续进化"倒推

**前提条件**：版本迭代机制必须有量化触发信号和自动执行路径。

**当前缺陷**：

4. **D4: 迭代触发仅有文字描述** — 4种触发信号（"做不了X"等）无法被程序化检测。缺少指标阈值配置（如"内容CTR连续2周低于2%触发迭代"）。

5. **D5: 无增量更新机制** — 当前只有"全量重建"路径，没有"增量修改"路径。用户说"加一个角色"需要重新走S1-S8全流程。

6. **D6: 知识沉淀与复用断裂** — protocol-knowledge-persistence 的存储位置（§5.5/§7.3）是文档章节而非可查询的知识库，新生成的专家包无法自动检索历史决策。

**改进优先级**：P1（影响长期价值，但不阻塞首次交付）

#### 第三步倒推：从"合规安全"倒推

**前提条件**：所有合规协议必须可自动执行、可审计、可追溯。

**当前缺陷**：

7. **D7: 合规敏感词列表硬编码** — S2 中的敏感词注册表是内联示例（medical/finance/education 各5个词），不是可扩展、可更新的外部知识库。

8. **D8: 合规检查点调用爆炸** — S7 同时调用10个协议，导致：①Token消耗巨大（每个协议都有完整SKILL.md上下文）；②检查顺序无优先级，可能先做低优先级检查浪费资源；③错误累积难以定位根因。

9. **D9: 数据安全评估依赖前置信息不完整** — protocol-data-security 需要 tools 列表和 domain_sensitivity，但 S6 的工具匹配结果和 S2 的合规激活映射之间没有直接数据管道，依赖 S7 手动桥接。

**改进优先级**：P0（合规红线，不可降级）

#### 第四步倒推：从"最小交互成本"倒推

**前提条件**：用户在首次构建中回答的问题数量最少、质量最高。

**当前缺陷**：

10. **D10: S1 问题数量固定不灵活** — Q1-Q9 的9个问题对所有用户一视同仁，但快速通道用户（如"帮我做一个美食号"）可能只需要3个问题就能确定需求，当前却要问完全部9题或走复杂的提前退出逻辑。

11. **D11: 快速通道跳过逻辑分散** — 快速通道的跳过逻辑分散在 S3（跳过合并入S4）、S4（精简确认流程）、S6（纯提示词方案）、S7（快速通道简化版）、S8（快速通道创建）中，缺少统一的快速通道编排。

12. **D12: 歧义消解"追问一次"策略过于机械** — S2 的四维消解模板中，当用户回答"都行"时只追问一次然后取默认值，没有利用后续阶段的自然上下文来隐式消解。

**改进优先级**：P1（影响用户体验，但不影响核心功能）

### 3.3 缺陷依赖关系与修复优先级

```
D7(敏感词硬编码) ──→ D8(协议调用爆炸) ──→ D3(回滚不可操作)
       │                    │
       ↓                    ↓
D9(安全评估信息断层) ──→ D1(验证不可自动化) ──→ D2(适配器浅层化)
                                                  │
                                                  ↓
                            D4(迭代触发文字化) ──→ D5(无增量更新)
                                                  │
                                                  ↓
                            D6(知识沉淀断裂) ──→ D5(无增量更新)

D10(S1问题固定) ──→ D11(快速通道分散) ──→ D12(歧义消解机械)
```

**修复优先级排序**（考虑依赖关系）：

| 优先级 | 缺陷ID | 改进项 | 依赖 |
|--------|--------|--------|------|
| **P0-1** | D8 | 协议调用优化：从"全部调用"改为"按需激活" | 无 |
| **P0-2** | D7 | 敏感词/合规知识外部化 | 无 |
| **P0-3** | D9 | 建立S2↔S6合规数据管道 | D7 |
| **P0-4** | D1+D2 | 平台适配器深度化+验证自动化 | D8 |
| **P1-1** | D3 | 回滚机制操作化 | D8 |
| **P1-2** | D11 | 快速通道统一编排 | 无 |
| **P1-3** | D10 | S1问题动态化 | D11 |
| **P1-4** | D12 | 歧义消解延迟化 | D10 |
| **P1-5** | D4+D5 | 增量更新+迭代触发量化 | D3 |
| **P2-1** | D6 | 知识沉淀可查询化 | D5 |

## 四、改进方案详细设计

### 改进1: 协议按需激活引擎 (Protocol On-Demand Activation)

**解决问题**：D8 — 协议调用爆炸

**当前状态**：S7 硬编码调用全部10个协议，无论领域类型和通道。

**改进方案**：

引入 **协议激活矩阵** (Protocol Activation Matrix)，根据 `domain_type × channel × stage` 三维组合确定需要激活的协议子集。矩阵定义在外部文件 `knowledge/protocol-activation-map.json`：

```json
{
  "protocol_activation_matrix": {
    "A_fast_S7":    ["quality-gate", "output-format"],
    "A_standard_S7": ["quality-gate", "compliance-engine", "output-format", "naming-convention"],
    "A_strict_S7":   ["quality-gate", "compliance-engine", "data-security", "output-format", "naming-convention", "tool-integration"],
    "B_fast_S7":     ["quality-gate", "output-format"],
    "B_standard_S7": ["quality-gate", "compliance-engine", "output-format", "naming-convention", "tool-integration"],
    "B_strict_S7":   ["quality-gate", "compliance-engine", "data-security", "error-handling", "output-format", "naming-convention", "tool-integration", "human-approval"],
    "C_fast_S7":     ["quality-gate", "output-format"],
    "C_standard_S7": ["quality-gate", "compliance-engine", "output-format", "naming-convention", "tool-integration", "automation-trigger"],
    "C_strict_S7":   ["quality-gate", "compliance-engine", "data-security", "output-format", "naming-convention", "tool-integration", "human-approval"],
    "D_fast_S7":     ["quality-gate", "output-format"],
    "D_standard_S7": ["quality-gate", "output-format", "naming-convention", "automation-trigger", "tool-integration"],
    "D_strict_S7":   ["quality-gate", "compliance-engine", "data-security", "output-format", "naming-convention", "tool-integration", "human-approval"],
    "F_fast_S7":     ["quality-gate", "output-format", "naming-convention"],
    "F_standard_S7": ["quality-gate", "compliance-engine", "output-format", "naming-convention", "error-handling", "feedback-loop"],
    "F_strict_S7":   ["quality-gate", "compliance-engine", "data-security", "error-handling", "feedback-loop", "output-format", "naming-convention", "tool-integration", "human-approval", "knowledge-persistence"],
    "_fallback":      ["quality-gate", "output-format"]
  },
  "fallback_rule": "未命中key时使用_fallback最小集，混合型取所有domain_tag对应的协议并集后去重"
}
```

**具体改动**：

1. 新增 `core-protocol-router` 核心引擎Skill，负责根据三维参数查询激活矩阵并按优先级排序调用
2. S7 的执行逻辑从"调用全部协议"改为"调用 core-protocol-router.activate(domain, channel, stage)"
3. 协议调用顺序遵循优先级：compliance > data-security > quality-gate > naming > output-format > tool-integration > error-handling > feedback-loop > human-approval > knowledge-persistence

**验证方式**：对比 v1.4.0 与 v2.0.0 在 A型快速通道 场景下的协议调用次数，目标减少 ≥60%。

### 改进2: 合规知识外部化 (Compliance Knowledge Externalization)

**解决问题**：D7 — 敏感词/合规知识硬编码

### 改进3: 上下文背包机制 (Context Backpack)

**解决问题**：B4(状态传递断层) + D9(安全评估信息断层)

**当前状态**：跨阶段仅传递结构化字段（output_schema定义的），不传递决策理由、置信度、用户偏好等"软信息"。

**改进方案**：

在 core-state-management-engine 中新增 `context_backpack` 字段，随阶段快照一起传递。**V-1.2修正：context_backpack 仅存储"软信息"，不复制已有结构化字段。合规数据通过 S2 的 output_schema 直接传递，不存储第二份副本。**

```json
{
  "context_backpack": {
    "decision_rationales": [
      {"stage": "S2", "decision": "确认A型", "reason": "用户明确说'做小红书内容'", "confidence": "high"},
      {"stage": "S2", "decision": "未选F型", "reason": "无持续客户交互需求", "confidence": "medium"}
    ],
    "user_preferences": {
      "interaction_style": "简洁",
      "detail_level": "探索者",
      "ai_experience": "novice"
    },
    "confidence_accumulator": {
      "domain": 0.85,
      "deliverable": 0.7,
      "platform": 0.6
    },
    "negative_constraints": [
      {"stage": "S2", "constraint": "不要使用Dify平台", "reason": "用户没有Dify账号"}
    ]
  }
}
```

**数据管道设计原则（V-1.2修正）**：合规数据的正确传递方式是让 S6 在 input_schema 中显式声明 `s2_outputs.compliance_activation_map` 作为输入，由 `core-state-management-engine` 保证跨阶段数据可访问性，**不在 context_backpack 中重复存储**。

**具体改动**：

1. core-state-management-engine 新增 `context_backpack` 的读写API
2. S1→S2：传递 user_preferences + confidence_accumulator 初始值
3. S2→S6：合规数据通过 `s2_outputs.compliance_activation_map` 直接传递（由core-state-management-engine保证可访问），S6的input_schema新增 `s2_outputs` 作为必填输入
4. S5→S7：传递 decision_rationales（S7可根据历史决策优化专家包生成）

**验证方式**：S6 能够直接读取 S2 的 compliance_activation_map，无需 S7 手动桥接。

### 改进4: 快速通道统一编排 (Fast Track Orchestrator)

**解决问题**：D11 + D10 — 快速通道逻辑分散 + S1问题不灵活

**当前状态**：快速通道跳过逻辑分散在S3/S4/S6/S7/S8中，S1的9个问题对所有用户固定。

**改进方案（V-1.3+V-1.4修正）**：

1. **S0 作为 S1 的内部子步骤（路由器，非判定器）**：S0 不独立修改通道，通道判定权仍由 S1→S2 链路负责。S0 仅决定"跳过哪些阶段"：

   ```text
   # 在 S1 生成 initial_channel_hint 后，内部调用 fast_track_router
   FUNCTION s1_fast_track_router(s1_output):
       clarity_score = EVALUATE_INPUT_CLARITY(s1_output)
       IF clarity_score >= 0.7 AND s1_output.team_size <= 1:
           RETURN {"skip_stages": ["S3", "S6_detailed"]}
       ELIF clarity_score >= 0.4:
           RETURN {"skip_stages": []}
       ELSE:
           RETURN {"extended_probing": true}
   ```

2. **EVALUATE_INPUT_CLARITY 函数定义（V-1.4修正）**：
   ```text
   FUNCTION EVALUATE_INPUT_CLARITY(s1_output):
       // 基于4个维度的加权评分，每个维度 0/1 二元判断
       domain_explicitness = s1_output.need_portrait.领域方向 != "自定义" ? 1 : 0
       deliverable_explicitness = s1_output.need_portrait.期望成果 != "自定义" ? 1 : 0
       team_size_clarity = s1_output.need_portrait.团队规模 IS NOT EMPTY ? 1 : 0
       platform_clarity = s1_output.need_portrait.目标平台 NOT IN ["还没想好", "其他"] ? 1 : 0
       RETURN 0.3 * domain_explicitness
            + 0.3 * deliverable_explicitness
            + 0.2 * team_size_clarity
            + 0.2 * platform_clarity
   ```
   阈值：≥0.7 → fast，0.4-0.7 → standard，<0.4 → strict

3. S1 问题动态化：
   - **核心问题**（必问，3题）：核心需求？目标用户？团队规模？
   - **扩展问题**（按需，最多6题）：仅在核心问题回答模糊时追问
   - **条件问题**（领域触发）：仅在检测到强监管领域时问合规相关问题

4. 移除独立的 `pipeline-s0-fast-track-gate` Skill，改为在 S1 的详细执行逻辑中调用内部路由函数。

**验证方式**：A型快速通道用户从S1到S8的交互轮次从当前的 ~15轮 降低到 ≤8轮。

### 改进5: 增量更新机制 (Incremental Update Mechanism)

**解决问题**：D5 — 无增量更新

**当前状态**：任何变更都需要重新走S1-S8全流程。

**改进方案**：

1. 新增 `pipeline-s9-incremental-update` Skill，支持3种增量操作：
   - **角色增删**：修改S5输出 → 局部更新S6/S7 → 不影响其他角色
   - **交付物调整**：修改S4输出 → 级联更新S5/S6/S7 → 保留未影响部分
   - **平台迁移**：修改platform参数 → 仅重新执行L3适配 → 保留S1-S6

2. 增量更新的最小变更集计算：
   ```text
   FUNCTION compute_change_set(change_request, current_snapshot):
       affected_stages = IDENTIFY_AFFECTED_STAGES(change_request)
       FOR stage IN affected_stages:
           IF IS_LOCAL_CHANGE(stage, change_request):
               RE_EXECUTE(stage)  // 仅重执行受影响阶段
           ELSE:
               CASCADE_DOWNSTREAM(stage)  // 级联更新下游
       RETURN incremental_patch
   ```

3. 变更影响矩阵（V-1.5修正：新增"反向合规检查"列）：

   | 变更类型 | 正向影响阶段 | 反向合规检查 | 需要用户确认 |
   |---------|------------|-------------|------------|
   | 新增角色 | S5→S6→S7 | 检查新角色是否触发新的合规条件→如有，级联S2 | 是（S5强确认） |
   | 删除角色 | S5→S6→S7 | 检查是否移除了关键合规角色→如有，级联S2 | 是（S5强确认） |
   | 修改交付物 | S4→S5→S6→S7 | 检查新交付物是否进入强监管领域→如有，级联S2 | 是（S4强确认） |
   | 切换平台 | L3 only | 检查新平台是否支持现有合规配置→如不支持，级联S7协议 | 是（Q9强确认） |
   | 更新合规规则 | S2→S6→S7 | 合规规则更新本身即级联，触发全链路合规重新评估 | 是（S2强确认） |

**验证方式**：用户说"加一个数据分析角色"时，仅需重新执行S5-S7，不需要从S1开始。

### 改进7: 确信度累计与跨阶段回退 (Confidence Accumulation & Cross-Stage Rollback)

**解决问题**：B1(需求-领域强耦合) + B2(S3/S4重叠)

**改进方案**：

1. 在 context_backpack 中引入 `confidence_accumulator`：

   ```text
   confidence_accumulator = {
       "domain": {"value": 0.85, "evidence": ["用户明确说做小红书", "关键词匹配度0.9"]},
       "deliverable": {"value": 0.7, "evidence": ["交付物明确为图文", "频率未确认"]},
       "architecture": {"value": 0.0, "evidence": []},  // S5前为0
       "toolchain": {"value": 0.0, "evidence": []}
   }
   ```

2. 每个阶段结束时更新确信度（V-1.6修正：校准增量和下降量）：
   - S1: domain 和 deliverable 初始值 = 0.5（最大不确定性）
   - S2: domain 确信度更新（消歧成功→+0.15，用户选择偏差→-0.2）
   - S5: architecture 确信度更新（MECE通过→+0.15）
   - S6: toolchain 确信度更新（工具可用→+0.1）
   - 平台兼容 →+0.1
   - 下降条件：领域重新分类→-0.3，合规冲突→-0.25

3. 确信度阈值触发行为（V-1.6修正：避免边界卡点）：
   - <0.4: 强制人工确认节点
   - 0.4-0.7: 标注⚠️+人工确认
   - >0.7: 自动通过

4. 跨阶段回退路径：
   ```text
   S8发现问题 → 评估根因阶段 → 直接回退到该阶段
   例如: S8验证发现角色协作不通 → 根因在S5架构 → 回退到S5（保留S1-S4产出）
   ```

**验证方式**：S8验证失败时，系统能自动识别根因阶段并回退，而非固定回退到S7。

### 改进8: 约束冲突仲裁机制 (Constraint Conflict Resolution)

**解决问题**：B6 — 31条规则可能冲突

**改进方案**：

1. 建立约束优先级矩阵：
   ```text
   优先级1 (法规级): 合规不可绕过(规则3) > 数据安全强制(规则11) > 紧急终止合规(规则20)
   优先级2 (平台级): 平台可执行强制(规则4) > 专家包格式合规(规则15) > 平台元数据完整(规则17)
   优先级3 (质量级): 交付物优先(规则1) > MECE强制(规则2) > 字段追踪(规则7)
   优先级4 (体验级): 单问规则(规则8) > 术语翻译(规则13) > 选项覆盖(规则14)
   ```

2. 冲突检测与仲裁规则：
   - 当规则A要求X但规则B禁止X时，取优先级更高的规则
   - 同优先级冲突时，取约束力更强的（强制>灵活）
   - 仲裁结果必须记录到 context_backpack.decision_rationales

3. 新增 `constraint-conflict-resolver` Skill：
   ```text
   FUNCTION resolve_conflict(rule_a, rule_b, context):
       priority_a = GET_PRIORITY(rule_a)
       priority_b = GET_PRIORITY(rule_b)
       IF priority_a != priority_b:
           RETURN HIGHER_PRIORITY_RULE
       ELSE:
           IF IS_MANDATORY(rule_a) AND NOT IS_MANDATORY(rule_b):
               RETURN rule_a
           ELSE:
               RETURN ARBITRATE_BY_CONTEXT(rule_a, rule_b, context)
   ```

**验证方式**：当规则2(MECE)与规则2灵活(角色精简)冲突时（3人团队要做5个角色的活），系统能自动仲裁出"角色精简优先，但必须保证第一责任人"。

## 六、不确定性应对策略

### 6.1 平台API不确定性

**问题**：不同平台的API/配置格式可能随时变化。

**策略**：
- 每个平台适配器增加 `api_version` 字段和 `compatibility_check` 步骤
- 适配器的配置模板使用"版本化模板"——不同API版本对应不同模板
- 当检测到API版本不匹配时，降级为"配置指导模式"（输出手动配置步骤而非自动配置文件）

### 6.2 用户需求不确定性

**问题**：用户可能在流程中途改变需求（如从A型改到F型）。

**策略**：
- 确信度累计机制自然支持——需求变更导致确信度下降，触发重新确认
- 新增"需求变更检测"逻辑：当用户在某阶段表达的意图与前序阶段不一致时，自动提示"您之前确认了X，现在似乎想做Y，是否需要重新确认？"

### 6.3 合规规则不确定性

**问题**：行业合规规则可能频繁更新。

**策略**：
- 合规知识外部化后，仅需更新JSON文件，不需要修改Skill代码
- 每次合规检查时添加"规则时效性标注"：`{"rule": "...", "as_of": "2026-06-17", "source": "...", "verify_required": true}`
- 强监管领域的合规规则标注"请核实最新版本"

### 6.4 角色协作不确定性

**问题**：多角色团队的实际协作效果可能不如预期。

**策略**：
- S5的SOP设计增加"协作磨合期"概念：前3次执行为"磨合期"，增加人工确认密度
- 反馈回路在磨合期内自动加密（从每阶段1次确认增加到每阶段2次）
- 磨合期结束后自动降频
